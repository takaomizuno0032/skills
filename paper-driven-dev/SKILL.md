# 論文駆動開発 (Paper-Driven Development)

論文の知見をコードベースに適用するためのワークフロー。

## 入力

ユーザーから以下を受け取る：

1. **論文**: PDFファイルパス または URL
2. **改善したいこと/実装したいこと**: 現在の課題や目標
3. **対象リポジトリ**: パス（引数で指定、省略時は現在のワーキングディレクトリ）

## フェーズ1: 論文の読解

論文を読み込み、以下を**日本語で**説明する：

### 1.1 論点 (Research Question)
- 論文が解決しようとしている問題は何か
- なぜその問題が重要なのか

### 1.2 方法 (Methodology)
- 提案されているアプローチ・手法
- 使用されているアルゴリズムや技術
- 実験設計

### 1.3 結論 (Findings)
- 主要な結果・発見
- 定量的な改善（あれば）
- 限界や今後の課題

## フェーズ2: 適用可能性の検討

対象リポジトリを**軽く探索**し、以下を検討する：

### 2.1 関連コードの特定
- 論文の手法が適用できそうな箇所を特定
- 現在の実装がどうなっているか概要を把握

### 2.2 実装可能性の評価
- 技術的に実装可能か
- 既存アーキテクチャとの整合性
- 必要な変更の規模感

**注意**: この段階では深い調査は行わない。追加調査が必要な場合はユーザーに確認する。

## フェーズ3: 提案

実装可能と判断した場合、以下を提案する：

### 3.1 実装方法
- 具体的なアプローチ
- 変更が必要なファイル・モジュール

### 3.2 期待される改善
- 論文に基づく改善予測
- 測定方法（可能であれば）

### 3.3 論文からの引用
- 提案の根拠となる論文の該当箇所を引用
- セクション番号や図表番号を明記

## フェーズ4: 実装

**ユーザーから「実装しましょう」等の明示的な指示があるまで実装に入らない。**

実装指示を受けたら、`pair-programming` スキルのプロセスに従う：

1. 最低3つの実装案を提案
2. ユーザーと議論して案を決定
3. 実装方針と手順を提案、了解を得る
4. 1コミット単位で段階的に実装

## 使用例

```
/paper-driven-dev ~/papers/attention-is-all-you-need.pdf ~/Projects/my-project

改善したいこと: 現在のRNNベースのモデルをTransformerに置き換えたい
```

```
/paper-driven-dev https://arxiv.org/abs/xxxx.xxxxx

改善したいこと: 推論速度を改善したい
対象リポジトリ: ~/Projects/ml-service
```
